# Delayed Proxies

Delayed proxies are a Bittensor wallet-security pattern where a proxy relationship includes a time
delay before a proxied action can execute. The delay creates a review window after the action is
announced, which gives the protected wallet holder time to inspect it and cancel it before it goes
through
([Delayed proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies),
[Proxy types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

Delayed proxies fit the same broader security model described in [[Coldkey and Hotkey Workstation
Security]]: keep the highest-authority coldkey in a safer environment and let a narrower,
internet-connected account handle routine work when possible.

## What the Delay Protects

The delay protects against immediate execution from a compromised or misused proxy account. A proxy
may still be allowed to submit a sensitive action, but the delay means the action waits through a
review period instead of executing at once. During that window, the delayed action can be canceled
before it takes effect
([Delayed proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

This is a timing control, not a full security guarantee. If nobody reviews pending actions, the
delay only postpones the problem.

## Distinction from Pure Proxies

[[Pure Proxies]] are keyless stand-in accounts used mainly to keep an address stable inside a wallet
structure such as a multisig arrangement. Delayed proxies are different: they are ordinary proxy
relationships where delegated actions wait before execution
([Delayed proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies),
[Proxy types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

In short, pure proxies solve account-structure continuity. Delayed proxies solve review time for
delegated authority.

## Distinction from Multisig Wallets

[[Multisig Wallets]] require a threshold of separate signers before an action can proceed. A delayed
proxy does not require multiple approvals. Instead, it lets one proxy relationship exist while
adding a waiting period before execution
([Delayed proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

In short, multisig changes who must approve. A delayed proxy changes when an approved proxy action
can execute and preserves a chance to cancel it during the waiting period.

## Reader Boundary

Delayed proxy should not be read as a replacement for coldkey custody, a substitute for
[[Multisig Wallets]], or proof that a wallet is safe without monitoring. It names a review-window
control on proxied actions. The protected coldkey still matters, the proxy scope still matters, and
the delay only helps if someone is available to review and cancel suspicious actions
([Delayed proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies),
[Proxy types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).
