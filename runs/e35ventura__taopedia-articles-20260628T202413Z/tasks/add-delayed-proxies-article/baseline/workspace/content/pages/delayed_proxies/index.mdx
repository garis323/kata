# Delayed Proxies

Delayed proxies are a Bittensor proxy-wallet security pattern that adds time between announcing a
proxied action and actually executing it. The key idea is not just delegation, but delay: the
protected wallet gets a review window before a sensitive action goes through. The official docs
describe delayed proxies alongside other proxy types and explain that the waiting period lets the
controller inspect what has been queued and react before execution
([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies),
[Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

This makes delayed proxies useful when a wallet owner wants proxy convenience without giving a proxy
account instant power over high-impact calls. They fit naturally beside [[Pure Proxies]],
[[Multisig Wallets]], and [[Coldkey and Hotkey Workstation Security]] as wallet-security concepts,
but they solve a different problem from each of those.

## What the Delay Protects

The delay protects against immediate execution from a compromised or misused proxy account. Instead
of letting the proxy submit a sensitive call and have it take effect at once, the delayed-proxy
model creates a review window before that action can be carried out. During that window, the
protected wallet holder can inspect the announced action and decide whether it is expected.

Just as important, the delayed action can be canceled during that waiting period. In other words,
the delay is not passive time; it is a chance to stop a queued proxied action before it executes.
That cancellation path is the practical security value of the pattern
([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

## Distinction from Pure Proxies

[[Pure Proxies]] are keyless stand-in accounts used mainly to keep wallet structure stable,
especially in multisig designs where membership may change. Delayed proxies are not about stable
identity. They are about adding a waiting period before a proxy action can execute
([Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

- **Pure proxies** keep an account structure stable without a direct signing key.
- **Delayed proxies** add a review-and-cancel window before proxied actions can complete.

The concepts are related because both sit in Bittensor's proxy vocabulary, but the security
question they answer is different: pure proxies answer continuity, while delayed proxies answer how
to reduce the risk of immediate delegated misuse.

## Distinction from Multisig Wallets

[[Multisig Wallets]] protect a wallet by requiring multiple signers to approve an action before it
can proceed. Delayed proxies protect a wallet by allowing delegated action only after a waiting
period, with the option to cancel during that period. One is threshold approval; the other is
time-based review.

- **Multisig wallets** require M-of-N signatory approval.
- **Delayed proxies** allow delegated execution only after a defined delay window.

That means delayed proxies do not replace multisig custody. A multisig answers "who must approve?"
while a delayed proxy answers "how much time is there to review and cancel before execution?"

## Reader Boundary

Delayed proxy should be read as a wallet-security control for delegated actions, not as a general
replacement for coldkey custody, [[Multisig Wallets]], or [[Pure Proxies]]. It narrows the risk of
delegation by creating a review window before execution and by allowing cancellation during that
window, but it still depends on the wallet owner noticing and responding to unwanted queued actions.
