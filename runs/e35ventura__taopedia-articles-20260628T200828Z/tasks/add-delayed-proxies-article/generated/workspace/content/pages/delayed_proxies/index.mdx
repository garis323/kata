# Delayed Proxies

Delayed proxies are a Bittensor wallet-security pattern that places time between a proxied request
and its execution. The official delayed-proxy guide describes that delay as a review window, so a
protected wallet owner can inspect what was queued before it takes effect and cancel it if the
request is not acceptable
([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies),
[Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

Delayed proxies are usually read alongside [[Pure Proxies]], [[Multisig Wallets]], and
[[Coldkey and Hotkey Workstation Security]] because all three topics describe different ways to
reduce the risk of exposing a primary coldkey during normal operations.

## What the Delay Protects

The delay protects against immediate execution by a proxy account that has already been granted
authority. If that proxy key is misused, the action does not land at once. Instead, the delay
creates a review window in which the primary authority can detect the queued call and cancel it
before execution
([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

That does not make the proxy harmless. The proxy can still request actions within its assigned
scope. The security gain is timing: sensitive proxied actions are no longer instant, which gives the
wallet owner time to review pending intent and intervene if needed
([Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

## Distinction from Pure Proxies

[[Pure Proxies]] and delayed proxies solve different wallet problems. A pure proxy is a keyless
stand-in account used for stable wallet structure, especially where membership or controlling keys
may change. A delayed proxy is an authority pattern for ordinary proxy delegation where execution is
intentionally postponed
([Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

In short, pure-proxy vocabulary is about stable account identity. Delayed-proxy vocabulary is about
adding a cancellation window before a proxied action can complete.

## Distinction from Multisig Wallets

[[Multisig Wallets]] protect a wallet by requiring multiple signers to approve an action. Delayed
proxies protect a wallet by allowing one delegated proxy to request an action but forcing that
request to wait before it executes
([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

These patterns can reduce different risks. Multisig raises the approval threshold. A delayed proxy
preserves delegated convenience while adding time for review and cancellation during the waiting
period.

## Reader Boundary

Readers should treat delayed proxies as a review-and-cancel control, not as a replacement for
careful key custody. The delay only helps if someone monitors pending requests and uses the window
to stop unauthorized actions when necessary, which is why the concept fits closely with
[[Coldkey and Hotkey Workstation Security]]
([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

The practical boundary is simple: delayed proxies create time to react, and the queued action can
be canceled during that time, but the proxy still needs narrow permissions and active oversight to
provide real wallet-security value.
