# Delayed Proxies

Delayed proxies are a Bittensor wallet-security pattern in which a proxy account can submit scoped
actions on behalf of a more sensitive wallet, but those actions do not execute immediately. Instead,
the configured delay creates a review window between announcement and execution. That gap gives the
owner time to inspect what was requested and, if necessary, cancel the delayed action before it
takes effect
([Delayed proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies),
[Proxy types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

This makes delayed proxies useful beside [[Coldkey and Hotkey Workstation Security]]. They let a
day-to-day workstation operate with narrower authority while keeping the primary wallet out of
routine signing flow.

## What the Delay Protects

The main protection is time. If a proxy account or its workstation is compromised, the attacker
does not get instant execution on delayed calls. The chain-enforced delay creates a review window in
which the wallet owner or monitoring team can notice the pending action, verify whether it is
expected, and cancel it before execution.

That means delayed proxies are not only about limiting which actions a proxy may attempt. They also
limit when those actions become final. The security value comes from combining scoped proxy
permissions with an opportunity to intervene after announcement but before execution.

## Distinction from Pure Proxies

[[Pure Proxies]] are keyless stand-in accounts used for wallet structure and continuity, especially
inside multisig arrangements where member slots may need to stay stable over time. Delayed proxies
solve a different problem: they are about delegated authority with a waiting period before
sensitive proxied actions can execute.

In short, pure proxies help preserve account structure, while delayed proxies help create a review
and cancellation window for delegated actions.

## Distinction from Multisig Wallets

[[Multisig Wallets]] protect a wallet by requiring a threshold of separate signers before an action
can go ahead. Delayed proxies protect differently: one authorized proxy can initiate a permitted
action, but the action waits through the configured delay before execution.

So the key distinction is the security control being applied. Multisig raises the number of required
approvals. A delayed proxy raises the amount of review time and makes cancellation possible during
that waiting period.

## Reader Boundary

Readers should treat delayed proxies as a wallet-risk reduction tool, not as automatic safety. A
delay only helps if someone monitors pending actions and acts in time when something looks wrong. If
nobody reviews the pending call, the delay may only postpone an unauthorized action instead of
stopping it.

For orientation, delayed proxies belong with Bittensor wallet-security topics such as
[[Coldkey and Hotkey Workstation Security]], not with subnet economics or validator scoring. The
core idea is simple: delayed proxies create a review window before execution, and the delayed action
can be canceled during that window.
