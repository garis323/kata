# Delayed Proxies

Delayed proxies are a Bittensor wallet-security pattern in which a proxy can submit a scoped action,
but the action does not execute immediately. The delay creates a review window before execution, and
the queued action can be canceled during that window if the request is wrong or the proxy account is
compromised ([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies),
[Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

This makes delayed proxies useful for sensitive day-to-day operations that should not require the
primary coldkey to stay online. They sit in the same wallet-security space as [[Pure Proxies]],
[[Multisig Wallets]], and [[Coldkey and Hotkey Workstation Security]], but they solve a different
problem.

## What the Delay Protects

The delay protects against immediate execution. If a proxy account submits a sensitive proxied call,
the coldkey owner or another authorized reviewer gets time to inspect it before it lands on-chain.
That review window matters because the action is pending rather than final, and the pending action
can be canceled before the delay expires ([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

In practice, the security gain is not that the proxy becomes trusted. The gain is that a compromised
or mistaken proxy does not get instant effect on delayed actions. The docs describe delay as part of
the control boundary for proxy types, so readers should treat it as a timing safeguard layered on
top of scoped permissions rather than as a replacement for careful key custody
([Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

## Distinction from Pure Proxies

[[Pure Proxies]] are keyless stand-in accounts used mainly for address continuity in wallet
structures such as multisig membership changes. Delayed proxies are not about stable account
identity. They are about delegated authority with a waiting period before execution
([Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types),
[Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

The difference is the failure mode each tool addresses. A pure proxy helps preserve structure when
the people or keys behind a wallet arrangement change. A delayed proxy helps reduce the risk of a
sensitive proxied action taking effect immediately.

## Distinction from Multisig Wallets

[[Multisig Wallets]] protect a wallet by requiring a threshold of signatory approvals before an
action can proceed. Delayed proxies protect a wallet differently: one proxy can submit an allowed
action, but the action waits through a review period and can still be canceled before execution
([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).

The distinction is approval threshold versus execution delay. Multisig answers "how many parties
must approve?" Delayed proxy answers "how much time exists to review or cancel a proxied action
before it executes?" Teams may use both patterns in a broader wallet design, but they are not the
same control.

## Reader Boundary

This article is about delayed proxies as a wallet-security concept, not a full setup guide. Readers
who need the broader workstation model should continue with [[Coldkey and Hotkey Workstation Security]].
Readers comparing proxy forms should also read [[Pure Proxies]] and the official proxy type
reference ([Proxy Types](https://docs.learnbittensor.org/keys/proxies/proxy-types)).

The practical takeaway is narrow: delayed proxies add a review window before sensitive proxied
actions execute, and that window allows cancellation before the action takes effect
([Delayed Proxies](https://docs.learnbittensor.org/keys/proxies/delayed-proxies)).
